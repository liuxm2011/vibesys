# VibeCoding 教学实践平台 — 产品需求文档 (PRD)

## 1. 项目概述

| 项目 | 信息 |
|------|------|
| **项目名称** | VibeCoding 教学实践平台 |
| **项目类型** | Web 应用（前后端分离） |
| **目标用户** | 高校计算机/软件工程专业学生、指导教师 |
| **核心理念** | 以 AI 驱动的开发流程实践，让学生体验从需求分析到代码生成的完整现代软件工程流程 |
| **开发模式** | VibeCoding — 通过结构化文档 + AI 编码工具完成项目开发 |

---

## 2. 项目背景与目标

### 2.1 背景

传统软件开发教学中，学生往往需要从零手写每一行代码。而在 AI 辅助编程时代，开发流程已经发生了根本性变化：开发者越来越多地使用结构化文档作为"提示工程"的输入，交由 AI 工具生成代码，然后进行审核和迭代。

本项目旨在让学生亲身体验这种新型开发范式，掌握以下核心能力：
- 将模糊的想法转化为结构化的产品需求文档（PRD）
- 设计清晰的前端组件架构与页面规范
- 设计严谨的后端 API 接口与数据模型
- 利用 AI 编码工具（Claude Code / Codex 等）进行高效开发

### 2.2 教学目标

| 目标 | 描述 |
|------|------|
| **需求分析能力** | 学生能够将选题转化为专业的 PRD 文档 |
| **系统架构设计** | 学生能够输出清晰的前端/后端技术文档 |
| **AI 工具使用** | 学生能够有效地使用 AI 编码工具将文档转为代码 |
| **工程实践** | 学生掌握现代 Web 开发的完整流程 |

## 3. 功能需求

### 3.1 整体流程

```
学生选题 → PRD 生成 → 前端文档 → 后端文档 → AI 编码工具开发 → 部署交付
  (Step 1)   (Step 2)    (Step 3)     (Step 4)      (Step 5)        (Step 6)
```

### 3.2 Step 1: 选题管理

**功能描述：** 学生从题库中选择或自拟项目选题。

| 功能点 | 详细说明 |
|--------|----------|
| 选题列表 | 展示所有可选项目，按难度分级（初级/中级/高级） |
| 选题详情 | 展示项目的简要描述、技术栈建议、预期功能模块 |
| 自定义选题 | 允许学生自拟选题，提交后由教师审核 |
| 选题锁定 | 学生选定后锁定选题，不可随意更换 |

**推荐选题示例：**
- 在线图书管理系统
- 个人博客平台
- 校园二手交易市场
- 在线考试系统
- 任务管理 / Todo 应用
- 餐厅在线点餐系统
- 电影信息聚合网站

### 3.3 Step 2: PRD 文档生成

**功能描述：** 基于选题，生成结构化产品需求文档。

**PRD 文档模板结构：**

```
PRD 文档
├── 1. 项目概述
│   ├── 项目名称
│   ├── 项目背景
│   ├── 目标用户
│   └── 核心价值
├── 2. 用户角色与权限
│   ├── 角色定义（如：普通用户、管理员）
│   └── 权限矩阵
├── 3. 功能需求
│   ├── 3.1 用户模块（注册/登录/个人信息）
│   ├── 3.2 核心业务模块
│   ├── 3.3 管理后台模块
│   └── 3.4 其他模块
├── 4. 非功能需求
│   ├── 性能要求
│   ├── 安全要求
│   └── 兼容性要求
├── 5. 页面清单
│   └── 列出所有需要开发的页面及简要描述
├── 6. 技术栈建议
│   ├── 前端技术栈
│   └── 后端技术栈
└── 7. 项目里程碑
    └── 开发阶段划分与时间规划
```

**要求：** PRD 必须足够详细，以便 AI 编码工具能够准确理解需求。

### 3.4 Step 3: 前端文档生成

**功能描述：** 基于 PRD，生成前端开发指导文档。

**前端文档模板结构：**

```
前端开发文档
├── 1. 技术栈说明
│   ├── 框架（React / Vue 等）
│   ├── UI 组件库
│   ├── 状态管理方案
│   ├── 路由方案
│   └── 构建工具
├── 2. 项目目录结构
│   └── 标准的前端项目文件组织
├── 3. 页面设计
│   ├── 每个页面的布局说明
│   ├── 组件拆分方案
│   └── 交互逻辑描述
├── 4. 组件清单
│   ├── 公共组件（Header, Footer, Sidebar 等）
│   ├── 业务组件
│   └── 组件 Props 定义
├── 5. 路由设计
│   └── 路由表（路径 / 组件 / 权限）
├── 6. API 调用规范
│   ├── 请求封装方式
│   ├── 错误处理
│   └── 数据格式约定
├── 7. 样式规范
│   ├── CSS 方案（Tailwind / CSS Modules 等）
│   ├── 设计规范（颜色、字体、间距）
│   └── 响应式策略
└── 8. 开发注意事项
    └── AI 编码工具使用提示
```

### 3.5 Step 4: 后端文档生成

**功能描述：** 基于 PRD，生成后端开发指导文档。

**后端文档模板结构：**

```
后端开发文档
├── 1. 技术栈说明
│   ├── 语言与框架（Node.js + Express / Python + FastAPI 等）
│   ├── 数据库选型（MySQL / PostgreSQL / MongoDB）
│   ├── ORM 工具
│   └── 认证方案（JWT / Session）
├── 2. 项目目录结构
│   └── 标准的后端项目文件组织
├── 3. 数据库设计
│   ├── ER 图
│   ├── 数据表清单
│   └── 字段定义（字段名、类型、约束、说明）
├── 4. API 接口文档
│   ├── 接口列表（URL、方法、权限）
│   ├── 请求参数（Body / Query / Path）
│   ├── 响应格式（成功/失败）
│   └── 示例请求与响应
├── 5. 业务逻辑
│   ├── 核心业务流程图
│   └── 关键算法说明
├── 6. 安全设计
│   ├── 认证与鉴权
│   ├── 数据校验
│   └── 防注入/XSS 策略
├── 7. 部署方案
│   ├── 环境配置
│   ├── 数据库迁移
│   └── 启动脚本
└── 8. 开发注意事项
    └── AI 编码工具使用提示
```

### 3.6 Step 5: AI 编码工具开发

**功能描述：** 学生将生成的文档提供给 AI 编码工具（如 Claude Code），进行代码生成。

**推荐的 AI 编码工具：**
| 工具 | 适用场景 |
|------|----------|
| Claude Code | 全栈代码生成，支持复杂逻辑 |
| OpenAI Codex CLI | 快速原型开发 |
| GitHub Copilot | 实时代码补全 |

**工作流程：**
1. 将前端文档 + PRD 交给 AI 工具，生成前端代码
2. 将后端文档 + PRD 交给 AI 工具，生成后端代码
3. 逐模块审查、调试、修复 AI 生成的代码
4. 前后端联调

### 3.7 Step 6: 部署交付

**功能描述：** 项目完成后，部署上线并提交交付物。

**交付物清单：**
- [ ] 完整的项目源代码（前端 + 后端）
- [ ] PRD 文档（终稿）
- [ ] 前端开发文档（终稿）
- [ ] 后端开发文档（终稿）
- [ ] 数据库设计文档
- [ ] API 接口文档（Swagger / Postman 集合）
- [ ] 部署说明文档
- [ ] 项目演示视频或截图

## 4. 非功能需求

### 4.1 文档质量要求

| 维度 | 要求 |
|------|------|
| **完整性** | 文档必须覆盖所有功能模块，不得有遗漏 |
| **准确性** | 接口定义、字段类型必须前后一致 |
| **可读性** | 使用 Markdown 格式，结构清晰，图表辅助说明 |
| **可执行性** | AI 编码工具能够根据文档直接生成可运行代码 |

### 4.2 评分标准（建议）

| 阶段 | 权重 | 评分维度 |
|------|------|----------|
| PRD 文档 | 20% | 完整性、逻辑性、专业性 |
| 前端文档 | 15% | 组件设计合理性、路由规范、样式规范 |
| 后端文档 | 15% | 数据库设计、API 规范、安全性 |
| 代码质量 | 30% | 功能完成度、代码规范、bug 数量 |
| 部署演示 | 10% | 能否正常运行、用户体验 |
| 答辩表现 | 10% | 对项目的理解、AI 工具使用心得 |

## 6. 项目里程碑

| 阶段 | 时间 | 交付物 |
|------|------|--------|
| **第一阶段：选题与 PRD** | 第 1-2 周 | 选题确认 + PRD 文档 |

## 7. 风险与应对

| 风险 | 影响 | 应对措施 |
|------|------|----------|
| AI 生成代码质量不高 | 项目进度延迟 | 加强文档质量，分模块逐步验证 |
| 文档与代码不一致 | 维护困难 | 每次修改代码同步更新文档 |
| 学生过度依赖 AI | 技术能力不足 | 要求逐行审查代码，提交代码审查报告 |
| 环境配置问题 | 开发受阻 | 提供统一的环境配置指南 |

### 8.1 AI 编码工具使用提示

**Prompt 编写建议：**

```
你是一个经验丰富的全栈开发工程师。请根据以下文档完成开发任务：

【PRD 文档】
（粘贴 PRD 相关内容）

【前端/后端文档】
（粘贴对应技术文档）

请按照以下步骤完成：
1. 先确认项目结构和依赖
2. 按模块逐步实现功能
3. 每个模块完成后进行自测
4. 遇到问题时先分析原因再修复
```
